## Demo Data Safety Statement

> **All data in this repository is 100% synthetic. No Protected Health Information (PHI)
> is stored, transmitted, or processed anywhere in this project.**

| Data Element | Demo Value | Real PHI? |
|---|---|---|
| Claim ID | CLM-0011, CLM-0012... | No — randomly generated |
| Member ID | M-10001, M-10002... | No — fictitious identifiers |
| Provider Name | Albany Medical Group, Peachtree Clinic... | No — generic names, no real NPI |
| Amount Billed | Random integer $500–$9,999 | No — randomly generated |
| Service Date | Random date 2024–2025 | No — randomly generated |
| Status | PAID / DENIED / PENDING | No — randomly assigned |
| City / State | Georgia cities | No — not linked to any real patient |

All claim documents are generated by `scripts/generate_claims.py` in this repository.
There are no real patient names, Social Security numbers, dates of birth, diagnoses,
procedure codes, or insurance member numbers anywhere in this project.

## Production HIPAA Controls

If this platform were deployed with real claims data, the following controls would be
required under the HIPAA Security Rule (45 CFR Part 164).

### 1. Encryption at Rest — 164.312(a)(2)(iv)

MarkLogic provides native Transparent Data Encryption (TDE). Enable it via the
Manage API on the `roxy-content` database:

```bash
curl --digest -u admin:password \
  -X PUT \
  -H "Content-Type: application/json" \
  --data '{"encryption-key-id": "AES-256"}' \
  http://localhost:8002/manage/v2/databases/roxy-content/properties
```

All data files on disk are encrypted at the storage layer. No application changes needed.

### 2. Encryption in Transit — 164.312(e)(1)

All MarkLogic ports must be secured with TLS certificates:

| Port | Service | TLS Action |
|---|---|---|
| 8000 | App Services / XQuery eval | Add SSL certificate |
| 8002 | Manage API | Add SSL certificate |
| 8040 | Claims REST endpoint | Add SSL certificate |
| 8041 | XDBC / MLCP ingest | Add SSL certificate |

Certificates are configured per App Server in the MarkLogic Admin UI or Manage API.

### 3. Role-Based Access Control — 164.312(a)(1)

MarkLogic has a built-in security role system. Minimum roles for a production claims
platform:

```xquery
xquery version "1.0-ml";
import module namespace sec = "http://marklogic.com/xdmp/security"
  at "/MarkLogic/security.xqy";

(: Read-only access to claims :)
sec:create-role("claims-reader", "Read-only access to claims data", (), (), ())

(: Write access for pipeline processing :)
sec:create-role("claims-writer", "Write access for pipeline transforms", (), (), ())

(: Admin access for configuration :)
sec:create-role("claims-admin",  "Full administrative access", (), (), ())
```

Users are assigned roles. No user has broader access than their role permits.

### 4. Audit Logging — 164.312(b)

Enable MarkLogic audit logging to capture every document access:

```bash
curl --digest -u admin:password \
  -X PUT \
  -H "Content-Type: application/json" \
  --data '{
    "audit-enabled": true,
    "audit-events": ["document-read", "document-insert", "document-update",
                     "document-delete", "login", "logout"]
  }' \
  http://localhost:8002/manage/v2/servers/Claims-App-Server/properties
```

Every read, write, and login is logged with timestamp, user, and document URI.

### 5. HIPAA-Eligible Hosting — 164.308(b)(1)

Docker must be deployed on a HIPAA-eligible infrastructure provider:

| Provider | HIPAA-Eligible Service |
|---|---|
| AWS | GovCloud, or standard regions with HIPAA BAA |
| Microsoft Azure | Azure Government, or standard with Business Associate Agreement |
| Google Cloud | Google Cloud Healthcare API, or standard with BAA |

Local development (laptop Docker) is acceptable for synthetic demo data only.

### 6. Business Associate Agreement — 164.308(b)(4)

Before loading any real PHI:

1. Sign a BAA with the cloud provider
2. Sign BAAs with any third-party vendors who access the system
3. Document all BAAs in the organisation's HIPAA compliance register

### 7. Minimum Necessary Access — 164.514(d)

SPARQL and XQuery endpoints are restricted by MarkLogic privilege:

```xquery
(: Restrict SPARQL endpoint to claims-reader role only :)
xdmp:security-assert(
  xdmp:permission("claims-reader", "execute"),
  "execute"
)
```

No endpoint returns more data than the requesting user's role permits.
